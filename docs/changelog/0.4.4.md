# [0.4.4] — Unreleased

Follow-up fixes discovered while re-verifying the 0.4.3 fixes end-to-end against a real cluster
(`docs/test-plan:1.1/`, `docs/report:1.1/`). Two of the 0.4.3 fixes (ISS-025 in particular) had scope
gaps that surfaced as new regressions once re-tested at the HTTP-route/capstone level rather than just
direct-repository unit tests.

### Fixed

- [ISS-028](../issues/ISS-028-cache-invalidation-key-mismatch.md) — cache invalidation on
  `save`/`update`/`updateForce`/etc. used a partition-key-only key that silently never matched
  `findById`'s real (full-key) cache key for any clustering-keyed `@CacheResult` entity, so writes
  never actually invalidated the stale cached row.
- [ISS-029](../issues/ISS-029-lookup-index-clustering-key-broken.md) — `@LookupIndex` resolution
  (`find`/`findAll`/`findPage`/`exists`/`deleteBy` via a lookup predicate) reconstructed only the
  primary table's partition key from the lookup row, never its clustering key, so it broke entirely
  (or, for `findPage`, silently scattered across the whole partition) for any entity combining a
  `@LookupIndex` with a clustering key, once `selectById` started requiring the full key (ISS-025).
