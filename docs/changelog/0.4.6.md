# [0.4.6] — 2026-07-23

### Added

- **`@GeneratedUuid(strategy: UuidStrategy = TIME_ORDERED)`** — auto-populates a `UUID` field on every
  INSERT, the same way `@CreatedAt` is auto-populated. Solves a real silent-data-loss risk: Cassandra/
  ScyllaDB writes are upserts on partition key + clustering key, so a clustering key derived from
  `Instant.now()` collides silently if two rows in the same partition land in the same millisecond —
  the second `save()` just overwrites the first, no exception, no warning.
  - `TIME_ORDERED` (default) — a UUIDv7 via the new `KandraUuid.timeOrdered()`. UUIDv7's 48-bit
    millisecond timestamp sits in the leading bytes, so plain lexicographic comparison — exactly how
    Cassandra's `UUIDType` comparator orders a generic `uuid` column — already sorts chronologically.
    No dependency on Cassandra's separate `timeuuid` CQL type, which Kandra doesn't map to at all.
  - `RANDOM` — a UUIDv4, for ids that must not leak their creation time.
  - Validated at schema-registration time like `@Version`: applying it to a non-`UUID` field throws
    `KandraSchemaException` immediately, not at first insert.
  - Never touches `update()`/`updateForce()` — primary key components are immutable in Cassandra.
- **`KandraUuid`** (`kandra-core`) — the UUIDv7/v4 generator backing `@GeneratedUuid`, also usable
  directly wherever application code needs a time-ordered or random id outside the entity insert
  lifecycle (`KandraUuid.timeOrdered()`, `KandraUuid.random()`).

### Changed

- Bumped `testcontainers` 1.19.8 → 1.21.4. 1.19.8 hardcodes its Docker API-version probe at `1.32`
  whenever the client config doesn't specify one; current Docker Engine releases have raised
  `MinAPIVersion` to 1.40 and reject that probe outright, failing every `kandra-test` integration test
  at container-startup, before any code under test even runs. 1.21.4 (same 1.x line, no API changes on
  our side) negotiates correctly. Dev/test dependency only — no runtime impact for consumers.

### Verification

`@GeneratedUuid` verified both at the unit level (`KandraUuidTest`: version/variant bits, cross-call
chronological sort order, uniqueness; `SchemaRegistryTest`: strategy collection, wrong-type validation)
and live against a real 3-node ScyllaDB cluster (`KandraIntegrationTest`): a `@GeneratedUuid @ClusteringKey`
column populates correctly on save ignoring any caller-supplied placeholder, and two rows written to the
same partition back-to-back land as two distinct, chronologically-ordered rows rather than one
overwriting the other.
